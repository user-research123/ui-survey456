### 数据完整性保障

1. **多竞品数据独立性**: 更新单日报告时，必须确保该日期下所有竞品（螃蟹、盼之、闲鱼）的数据完整性，避免因单独更新某一家竞品而覆盖或遗漏其他竞品的已有数据

2. **空状态处理**: 若某天某渠道无有效数据，报告中应保留该渠道区块但内容为空，严禁复制前一天数据或隐藏该区块

3. **板块独立性**: "竞品动态追踪"与"用户需求追踪"两个板块拥有各自独立的日期切换按钮与内容区块，互不干扰

4. **微博数据特殊性**: 微博舆情分析关注用户需求与声量，与竞品交易数据形成互补；若当天微博数据不足30条，如实报告实际数量并标注样本量较小

## 四、技术架构要点

### 浏览器自动化策略

| 平台 | 反爬机制 | 应对方案 |
|------|---------|---------|
| 螃蟹账号 (pxb7.com) | 瀑布流懒加载 | JS 注入滚动 + DOM 变化检测 |
| 盼之 (pzds.com) | 阿里云滑块验证码 | 浏览器自动化获取最新 Cookie 令牌 |
| 闲鱼 (goofish.com) | 登录态验证 + 动态渲染 | Cookie 注入 + JS 滚动加载 |
| 微博 (weibo.com) | 未登录数据缺失 + 反爬限制 | URL 参数分页 (`?page=N`) + 控制请求频率 + Cookie注入(可选) |

### Fallback 机制

- **盼之平台**: 当实时爬取受阻于验证码且多次尝试失败时，允许复用近期已成功获取的基准数据集（如最近一次的 100 个商品样本）进行趋势分析与报告生成
- **标注规范**: 在报告中明确标注数据来源为"基准数据集（Fallback）"
- **微博舆情**: 如Cookie过期导致无法获取作者、时间等字段，至少提取帖子正文内容进行分析；如数据量不足30条，如实报告并标注样本量较小

## 五、运维与监控

### 任务状态检查

```bash
# 查看所有定时任务
use_cron(action=list)

# 查看特定任务详情
use_cron(action=list, limit=5)
```

### 常见问题排查

1. **任务未执行**: 检查 `enabled` 状态是否为 `true`
2. **数据为空**: 检查 Fallback 策略是否生效，确认基准数据集是否存在
3. **GitHub 推送失败**: 检查 Git Token 是否过期，Remote URL 配置是否正确
4. **验证码拦截**: 检查浏览器自动化是否正常完成滑块验证，Cookie 令牌是否最新

### 手动触发测试

如需立即测试某个任务，可临时修改执行时间为当前时间后 1 分钟，或使用 `use_cron(action=run, job_id=<任务ID>)` 手动触发。
