# Auto Research 工作流程详细说明

## 整体架构

Auto Research 采用六步流水线设计，通过并行化和智能调度实现高效调研：

```
┌─────────────┐
│  主题输入    │
└──────┬──────┘
       ↓
┌─────────────┐
│  拆分问题    │ ← 将主问题拆分为3-5个子问题
└──────┬──────┘
       ↓
┌─────────────────────┐
│  迭代搜索            │ ← Task子代理并行搜索(2-3组并发)
│  (追踪知识缺口)      │ ← WebFetch + 浏览器自动化降级
└──────┬──────────────┘
       ↓
┌─────────────────────┐
│  多视角分析          │ ← Task并行分析(技术/业务/竞争/风险)
│                      │ ← 交叉验证 + 矛盾识别
└──────┬──────────────┘
       ↓
┌─────────────────────┐
│  场景压力测试        │ ← 边界条件/异常场景/演进场景
└──────┬──────────────┘
       ↓
┌─────────────┐
│  生成报告    │ ← 结构化Markdown +可选钉钉发布
└─────────────┘
```

## Step 1: 主题输入

### 输入格式

```bash
/research Topic: <调研主题> --depth <standard|deep> --publish<true|false>
```

### 参数说明

- **Topic**（必填）：调研主题，应具体明确
  - ✅ 好例子："KV Cache 在推理集群中的通信模式"
  - ❌ 坏例子："AI技术"（过于宽泛）

- **--depth**（可选）：调研深度
  - `standard`（默认）：标准深度，适合一般性调研，耗时约8-10分钟
  - `deep`：深度调研，覆盖更多维度和来源，耗时约12-15分钟

- **--publish**（可选）：是否发布到钉钉文档
  - `false`（默认）：仅生成本地 Markdown 文件
  - `true`：生成后自动发布到钉钉文档

### 主题解析

系统会解析主题，提取：
- 核心研究对象（如"KV Cache"）
- 研究维度（如"通信模式"）
- 应用场景（如"推理集群"）

## Step 2: 拆分问题

### 拆分原则

1. **覆盖核心维度**：确保涵盖技术原理、应用场景、优缺点对比、最佳实践等
2. **相对独立**：子问题之间尽量独立，便于并行搜索
3. **具体明确**：每个子问题足够具体，避免过于宽泛
4. **数量适中**：通常3-5个子问题，deep模式下可增加至5-7个

### 示例

**主题**："KV Cache在推理集群中的通信模式"

**拆分结果**：
1. KV Cache 的基本原理和存储结构是什么？
2. 推理集群中 KV Cache 的常见通信模式有哪些？
3. 不同通信模式的性能对比和适用场景？
4. 业界最佳实践和优化方案？

**Deep模式额外问题**：
5. KV Cache 的技术实现细节和架构设计？
6. 主要竞品的技术方案对比？

### 实现逻辑

```python
def split_questions(topic: str, depth: str) -> List[str]:
    """智能拆分问题"""
    
    # 基础维度（standard模式）
    base_questions = [
        f"{topic} 的基本原理和核心概念是什么？",
        f"{topic} 的主要应用场景和典型案例？",
        f"{topic} 的优缺点分析和局限性？",
        f"{topic} 的行业最佳实践和发展趋势？"
    ]
    
    # 深度维度（deep模式额外添加）
    if depth == "deep":
        deep_questions = [
            f"{topic} 的技术实现细节和架构设计？",
            f"{topic} 的竞品对比和差异化优势？"
        ]
        return base_questions + deep_questions
    
    return base_questions
```

## Step 3: 迭代搜索（追踪知识缺口）

### 并行搜索架构

QoderWork 版的核心优势是并行化搜索：

```
主协调器
  ├─ Task子代理1 → 子问题1 + 子问题2
  ├─ Task子代理2 → 子问题3 + 子问题4
  └─ Task子代理3 → 子问题5（deep模式）
```

### 搜索策略

#### 第一轮搜索：广泛收集

对每个子问题进行广泛搜索，目标：
- 收集5-10个相关来源
- 覆盖不同类型的资源（官方文档、技术博客、学术论文、行业报告）
-记录每个来源的相关性评分

####知识缺口识别

分析已获取信息，识别缺失的关键点：

**检查维度**：
- 定义/原理：是否有清晰的定义和原理解释？
- 应用场景：是否有具体的应用案例？
- 优缺点：是否有全面的优劣势分析？
- 数据支撑：是否有量化数据或实证研究？
- 来源多样性：是否有至少3个不同来源？

**缺口示例**：
- "缺少'性能数据'维度的信息"
- "来源不足（当前2个，建议至少3个不同来源）"
- "缺少'实际应用案例'的具体描述"

####第二轮搜索：定向补充

针对识别的知识缺口进行定向搜索：
- 调整搜索关键词，聚焦缺失维度
-优先选择高权威性来源
- 限制迭代次数（通常最多2轮），避免无限循环

### 内容抓取策略

#### 优先级1: WebFetch（web-content-fetcher skill）

使用 URL 前缀转换服务快速获取结构化 Markdown：
- `https://markdown.new/<url>`
- `https://defuddle.md/<url>`
- `https://r.jina.ai/<url>`

**优势**：速度快、结构化好、反爬能力强

#### 优先级2: 浏览器自动化降级

当 WebFetch 失败时，自动切换为浏览器自动化：
- 使用 `use_browser` 打开页面
-模拟滚动加载（针对瀑布流页面）
- 使用 `readability` 提取正文内容

**适用场景**：
- 动态渲染页面
- 需要登录的页面
- 反爬严格的网站

### TodoWrite 进度反馈

实时显示调研进度：

```
✓子问题1: KV Cache 基本原理 [已完成 - 5个来源]
✓ 子问题2: 通信模式分类 [已完成 - 4个来源]
- 子问题3: 性能对比分析 [进行中 - 第二轮搜索]
  ├─ 第一轮: 3个来源
  └─ 知识缺口: 缺少量化数据
○ 子问题4: 最佳实践 [待开始]
```

## Step 4: 多视角分析

### 分析维度

启动 Task 子代理并行执行四个视角的分析：

#### 1. 技术视角

**关注点**：
- 技术原理和架构设计
- 核心算法和实现细节
- 性能指标和优化空间
- 技术成熟度和稳定性

**输出**：
- 技术架构图解
- 关键性能数据
- 技术选型建议

#### 2. 业务视角

**关注点**：
- 应用场景和商业价值
- 成本效益分析（TCO、ROI）
-市场接受度和 adoption rate
- 商业模式和盈利路径

**输出**：
- 商业价值评估
- 成本对比表格
- 市场推广建议

#### 3. 竞争视角

**关注点**：
- 竞品对比和差异化优势
- 市场份额和竞争格局
- 进入壁垒和护城河
- 合作伙伴和生态系统

**输出**：
- 竞品对比矩阵
- SWOT分析
- 竞争策略建议

#### 4. 风险视角

**关注点**：
- 技术风险和局限性
- 市场风险和不确定性
- 合规风险和政策影响
- 供应链和依赖风险

**输出**：
- 风险清单和等级评估
- 缓解措施建议
- 应急预案

### 交叉验证

#### 一致发现识别

找出多个视角都提到的关键发现，这些通常是核心结论：

```python
# 示例：发现"性能优化"在技术视角和业务视角都被提及
consistent_findings = [
    {
        "finding": "性能优化是关键考量因素",
        "mentioned_by": ["技术视角", "业务视角"],
        "confidence": "high"
    }
]
```

#### 矛盾识别

发现不同来源之间的矛盾点并标注：

```python
contradictions = [
    {
        "topic": "性能提升幅度",
        "source_a": "来源A声称提升50%",
        "source_b": "来源B声称提升30%",
        "resolution": "需进一步验证，可能与测试环境有关"
    }
]
```

#### 证据分级

根据来源权威性和一致性对证据进行分级：

- **高置信度**：多个权威来源一致 + 有量化数据支撑
- **中置信度**：2-3个来源一致或有部分数据支撑
- **低置信度**：单一来源或缺乏数据支撑

## Step5: 场景压力测试

### 测试场景设计

#### 1. 边界条件测试

**场景描述**：极端条件下的表现

**测试维度**：
- 数据量极大/极小
- 并发极高/极低
- 网络延迟极高
-资源受限环境

**风险评估**：
- 性能瓶颈暴露
- 资源消耗超出预期
- 稳定性下降

**缓解措施**：
- 提前进行压力测试
- 设计弹性扩容方案
- 设置监控告警阈值

#### 2. 异常场景测试

**场景描述**：故障和异常情况下的表现

**测试维度**：
- 网络中断
- 节点故障
- 数据不一致
- 依赖服务不可用

**风险评估**：
- 服务可用性受影响
- 数据一致性难以保证
- 恢复时间可能较长

**缓解措施**：
- 设计容错和降级机制
- 实施数据备份策略
- 制定应急预案

#### 3. 演进场景测试

**场景描述**：长期演进中的适应性

**测试维度**：
- 技术升级和迁移
- 需求变化和功能扩展
- 规模扩张（10x、100x）
- 团队和组织变化

**风险评估**：
-技术债务积累
- 架构无法支撑
- 迁移成本很高

**缓解措施**：
- 保持架构灵活性
- 定期技术评审和重构
- 预留扩展接口

### 风险等级评估

根据风险数量和严重程度评估风险等级：

- **高风险**：≥3个严重风险或存在单点故障
- **中风险**：2个中等风险或部分风险有缓解措施
- **低风险**：≤1个轻微风险且有完善缓解措施

##性能优化

### QoderWork 版 vs Claude Code 版

| 特性 | Claude Code 版 | QoderWork 版 | 提升 |
|------|----------------|--------------|------|
| 搜索方式 | 顺序迭代 | Task子代理并行（2-3组并发） | 2-3x |
| 进度反馈 | 文本打印 | TodoWrite实时进度条 | 用户体验 |
| 页面抓取 | WebFetch | WebFetch + 浏览器自动化降级 | 成功率↑ |
| 分析方式 |顺序执行 | Task并行分析 | 1.5-2x |
| 报告发布 |本地Markdown | 本地 + 可选钉钉文档一键发布 | 功能增强 |
| 总耗时 | ~15分钟 | ~8-10分钟 | **30-40%** |

### 并行化策略

**Task子代理分配**：
- 子问题较少（≤4个）：2个子代理，每个负责2个子问题
- 子问题较多（5-7个）：3个子代理，负载均衡分配

**并行分析**：
- 四个视角（技术/业务/竞争/风险）同时启动
- 各自独立分析，最后汇总交叉验证

### 缓存策略

**搜索结果缓存**：
- 相同查询结果缓存24小时
- 避免重复搜索相同内容

**页面内容缓存**：
- 抓取的页面内容缓存7天
- 减少重复抓取开销

## 错误处理

### 常见问题及解决方案

#### 1. 搜索结果为空

**原因**：关键词过于具体或冷门

**解决**：
- 扩展关键词范围
- 尝试同义词和相关术语
- 降低相关性阈值

#### 2. 页面抓取失败

**原因**：反爬严格、需要登录、动态渲染

**解决**：
- 自动降级为浏览器自动化
- 尝试不同的 URL 前缀转换服务
- 记录失败原因供后续优化

#### 3. 知识缺口无法填补

**原因**：信息确实稀缺或关键词不当

**解决**：
- 标注"信息不足"并在报告中说明
- 建议用户手动补充
- 调整搜索策略重新尝试

####4. 钉钉发布失败

**原因**：API凭证失效、权限不足、网络问题

**解决**：
- 检查 API 凭证配置
- 确认文档创建权限
- 重试或降级为仅本地保存

## 最佳实践

### 主题选择

✅ **好的主题**：
- 具体明确："KV Cache 在推理集群中的通信模式"
- 有研究价值：新技术、新趋势、决策支持
- 范围适中：不太宽泛也不太狭窄

❌ **不好的主题**：
- 过于宽泛："AI技术"
- 过于狭窄："某个特定版本的某个函数的某行代码"
- 缺乏研究价值："今天天气如何"

### 深度选择

**standard 模式**适用于：
-快速了解一个领域
- 日常技术调研
- 初步可行性分析

**deep 模式**适用于：
- 重要决策支持
- 深度竞品分析
-学术研究综述

### 发布策略

**仅本地保存**：
- 个人学习和研究
-草稿和初步调研
- 敏感信息不宜公开

**钉钉发布**：
- 团队共享和协作
- 正式调研报告
- 需要存档和追溯
