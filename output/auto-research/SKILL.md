# Auto Research - 自动化调研技能

## 概述

Auto Research 是一个调研自动化技能，通过并行化搜索和多视角分析，快速生成结构化的 Markdown 调研报告。每个论断都有出处引用，禁止臆造。

**核心优势：**
- **并行加速**：Task 子代理并行搜索（2-3组并发），整体提速约 30-40%
- **实时进度**：TodoWrite 实时显示调研进度
- **智能降级**：WebFetch + 浏览器自动化双策略确保内容获取成功率
- **一键发布**：支持本地 Markdown 保存和钉钉文档一键发布
- **严格引用**：每个结论都有明确出处，保证报告可信度

## 使用方式

### 基本用法

```
/research Topic: <调研主题> --depth <standard|deep> --publish <true|false>
```

**参数说明：**
- `Topic`: 调研主题（必填）
- `--depth`: 调研深度
  - `standard`（默认）：标准深度，适合一般性调研
  - `deep`：深度调研，覆盖更多维度和来源
- `--publish`: 是否发布到钉钉文档
  - `false`（默认）：仅生成本地 Markdown 文件
  - `true`：生成后自动发布到钉钉文档

### 示例

```bash
# 标准深度调研，不发布
/research Topic: KV Cache 在推理集群中的通信模式 --depth standard

# 深度调研，发布到钉钉文档
/research Topic: QUIC 协议在推理服务中的应用 --depth deep --publish true

# 自然语言触发（QoderWork）
调研一下 QUIC 协议在推理服务中的应用 --depth deep --publish
```

## 工作流程

Auto Research 采用六步流水线，全程自动化执行：

```mermaid
graph LR
    A[主题输入] --> B[拆分问题]
    B --> C[迭代搜索<br/>追踪知识缺口]
    C --> D[多视角分析]
    D --> E[场景压力测试]
    E --> F[生成报告]
```

### Step 1: 主题输入

接收用户提供的调研主题，理解核心问题和边界条件。

**输入示例：**
- "KV Cache 在推理集群中的通信模式"
- "QUIC 协议在推理服务中的应用"
- "2024年AI芯片市场格局分析"

### Step 2: 拆分问题

将主问题拆分为 3-5 个子问题，确保覆盖核心维度。

**拆分原则：**
- 覆盖技术原理、应用场景、优缺点对比、最佳实践等维度
- 子问题之间相对独立，便于并行搜索
-每个子问题足够具体，避免过于宽泛

**示例（以"KV Cache 通信模式"为例）：**
1. KV Cache 的基本原理和存储结构是什么？
2. 推理集群中 KV Cache 的常见通信模式有哪些？
3. 不同通信模式的性能对比和适用场景？
4. 业界最佳实践和优化方案？

### Step 3: 迭代搜索（追踪知识缺口）

**QoderWork 版特性：并行搜索**

启动 2-3 个 Task 子代理并行执行搜索任务，每个子代理负责 1-2 个子问题。

**搜索策略：**
1. **第一轮搜索**：针对每个子问题进行广泛搜索，收集初步资料
2. **知识缺口识别**：分析已获取信息，识别缺失的关键点
3. **第二轮搜索**：针对知识缺口进行定向搜索
4. **迭代优化**：重复步骤 2-3，直到信息饱和度达到阈值

**工具链：**
-优先使用 `web-content-fetcher` skill 获取网页内容
- WebFetch 失败时自动降级为浏览器自动化抓取
- 所有搜索结果记录来源 URL 和关键摘要

**TodoWrite进度反馈：**
```
✓ 子问题1: KV Cache 基本原理 [已完成]
✓子问题2: 通信模式分类 [已完成]
- 子问题3: 性能对比分析 [进行中 - 第二轮搜索]
○ 子问题4: 最佳实践 [待开始]
```

### Step 4: 多视角分析

启动 Task 子代理并行执行多维度分析：

**分析维度：**
1. **技术视角**：技术原理、架构设计、实现细节
2. **业务视角**：应用场景、商业价值、成本效益
3. **竞争视角**：竞品对比、优劣势分析、差异化定位
4. **风险视角**：潜在风险、限制条件、应对策略

**分析方法：**
- 交叉验证：多个来源的信息相互印证
- 矛盾识别：发现不同来源之间的矛盾点并标注
- 证据分级：区分事实陈述、专家观点、推测判断

### Step 5: 场景压力测试

对分析结论进行压力测试，验证其在极端场景下的鲁棒性：

**测试场景：**
- **边界条件**：数据量极大/极小、并发极高/极低
- **异常场景**：网络中断、节点故障、数据不一致
- **演进场景**：技术升级、需求变化、规模扩张

**输出形式：**
- 标注每个结论的适用条件和限制
- 识别高风险假设和不确定性因素
- 提供备选方案和应急预案

### Step 6: 生成报告

生成结构化的 Markdown 报告，包含以下章节：

#### 报告结构

```markdown
# [调研主题]

## TL;DR（摘要）

用 3-5 句话总结核心发现和关键建议。

## 关键发现

### 发现1: [标题]
- 详细描述
- 支撑证据（带引用）

###发现2: [标题]
...

## 对比分析

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
|性能 | ...   | ...   | ...   |
| 成本 | ...   | ...   | ...   |

## 建议

基于分析结果，提供 2-4 条 actionable 的建议。

## 参考文献

1. [标题](URL) - 来源说明
2. [标题](URL) - 来源说明
...
```

**引用规范：**
- 每个论断必须标注来源（使用脚注或内联引用）
- 优先引用权威来源（官方文档、学术论文、行业报告）
- 避免单一来源依赖，重要结论需多方印证

## 资源说明

本技能包含以下辅助资源：

### scripts/

- `research_orchestrator.py` - 主协调器，负责任务拆分、并行搜索调度、结果汇总

- `report_generator.py` - 报告生成器，生成结构化 Markdown 报告

### references/

- `workflow.md` - 完整工作流程详细说明
- `output_template.md` - 报告输出模板和格式规范

## 性能指标

**QoderWork 版 vs Claude Code 版：**

| 特性 | Claude Code 版 | QoderWork 版 |
|------|----------------|--------------|
| 搜索方式 | 顺序迭代 | Task 子代理并行（2-3组并发） |
| 进度反馈 | 文本打印 | TodoWrite 实时进度条 |
| 页面抓取 | WebFetch | WebFetch + 浏览器自动化降级 |
| 分析方式 | 顺序执行 | Task 并行分析 |
| 报告发布 | 本地 Markdown | 本地 + 可选钉钉文档一键发布 |
| 耗时 | ~15 分钟 | ~8-10分钟（快 30-40%） |

## 注意事项

1. **严格引用**：所有结论必须有明确出处，禁止臆造
2. **知识缺口追踪**：持续识别信息盲区，确保调研完整性
3. **并行效率**：充分利用 Task 子代理并行能力，缩短总耗时
4. **降级策略**：WebFetch 失败时自动切换浏览器自动化，提高成功率
5. **发布确认**：钉钉文档发布前需用户确认，避免误操作

## 典型使用场景

- **技术调研**：新技术选型、架构设计参考、最佳实践收集
- **市场分析**：竞品分析、行业趋势、市场规模评估
- **产品研究**：用户需求洞察、功能对比、定价策略
- **学术研究**：文献综述、研究现状梳理、前沿动态跟踪
